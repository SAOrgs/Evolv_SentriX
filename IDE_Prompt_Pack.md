# IDE prompt pack — Compound Risk Intelligence Platform

How to use this document: paste **Prompt 0** into both of your IDE assistants first (Copilot, Antigravity, Claude Code, whatever you're each using) so both tools share the same project understanding. Then each person follows their own track (Person A or Person B), pasting the numbered prompts in order as you go. Merge checkpoints are marked — stop and integrate at those points before continuing.

**Recommended strategy: staged parallel work, not one big-bang merge at the end.** Two people building fully independently for 20 days and merging once on day 19 is the highest-risk option — integration bugs surface with no time left to fix them. Instead: agree the API contract on day 1–2 (below), build independently against that contract for a week, do a real integration checkpoint, then split again. This is the split reflected in the prompts below.

**Branching convention:**
- `main` — always demo-able, only merge tested code here
- `feature/backend-agents` — Person A's branch
- `feature/frontend-dashboard` — Person B's branch
- Merge to `main` at each checkpoint (days 9, 16, 19), not continuously

## Prompt 0 — shared project context (paste into both IDEs first)

```
We are building a hackathon prototype called the "Compound Risk Intelligence
Platform" for an industrial worker-safety hackathon (theme: Industrial
Intelligence / Worker Safety / Geospatial Safety Analytics). We have 20 days
and a team of two people working on separate git branches.

PROBLEM: Indian industrial plants already have gas/IoT sensors, SCADA,
CCTV, permit-to-work systems, and maintenance logs — but each system works
in isolation. Nobody correlates them. A hazardous condition often only
becomes visible when two or three individually-normal signals combine
(e.g. a hot work permit is active in a zone where gas readings are
trending upward and no fire watch has been logged). We are building the
missing correlation layer.

WHAT WE ARE BUILDING (scope for this hackathon — do not expand beyond this):
1. A synthetic data generator producing realistic gas/IoT sensor streams,
   permit records, and maintenance/shift logs (we have no real plant data
   access — this is explicitly synthetic and we say so in the demo).
2. A FastAPI backend with three domain agents:
   - Gas/SCADA agent: detects threshold breaches and trending anomalies
   - Permit agent: detects conflicting/dangerous simultaneous operations
     (e.g. hot work permit near elevated gas readings)
   - Maintenance/shift agent: flags overdue maintenance tasks and fatigue-
     risk shift patterns
3. An orchestrator/fusion agent that looks for CO-OCCURRING conditions
   across the three agents' outputs that would look benign individually,
   and produces a single 0-100 compound risk score per plant zone, with a
   plain-language explanation of which signals combined and why.
4. A regulatory RAG module: a small vector store over real OISD/DGMS/
   Factory Act clauses plus a handful of synthetic near-miss reports, so
   every alert cites the specific regulation it relates to.
5. A React dashboard showing:
   - A geospatial heatmap (a georeferenced SVG plant floorplan is enough —
     no real GIS needed) with live risk zones and active permits overlaid
   - A risk-score timeline (recharts)
   - An explainable alert panel showing the trigger trail and regulatory
     citation for each alert
6. An emergency-response flow — for this hackathon this is MOCKED, not
   fully built: one canned "evacuation alert + auto-drafted incident
   report" output triggered when risk score crosses a threshold. Do not
   spend real engineering time on live SMS/PA integrations.
7. CCTV/computer-vision is OUT OF SCOPE for the build — mention only as a
   future data source on the architecture slide. Do not build any CV code.

TECH STACK (fixed — do not suggest alternatives unless something is
genuinely broken):
- Backend: Python + FastAPI
- Agent orchestration: LangGraph if familiar, otherwise a simple hand-
  rolled state machine — do not spend build time learning a new framework
- Vector store: Chroma or FAISS
- LLM: Claude API (model: claude-sonnet-4-6) for the orchestrator's
  reasoning/explanation and the RAG's answer synthesis
- Frontend: React + Tailwind CSS, recharts for charts
- No database server required for the hackathon — SQLite or in-memory
  state is fine; we are not optimizing for production scale
- Data: 100% synthetic, generated by us. Never imply real plant validation
  anywhere in code comments, docs, or UI copy.

TEAM SPLIT AND API CONTRACT (agree this now, before writing feature code):
- Person A owns: synthetic data generator, ingestion, the three domain
  agents, the orchestrator/fusion agent, the regulatory RAG. Branch:
  feature/backend-agents
- Person B owns: the React dashboard — geospatial heatmap, risk-score
  timeline, alert panel, and overall UI/UX. Branch:
  feature/frontend-dashboard

Both sides depend on this shared API contract. Backend (Person A) must
implement it; frontend (Person B) builds against a mocked version of it
until the real backend is ready, then swaps the mock for the real
endpoint with no other UI changes needed.

  GET  /api/zones                -> list of plant zones with coordinates
                                     for the floorplan
  GET  /api/risk-scores           -> current risk score (0-100) per zone,
                                     with a timestamp
  GET  /api/risk-scores/history?zone_id=&hours=  -> time series for the
                                     timeline chart
  GET  /api/alerts                -> list of active/recent alerts, each
                                     with: zone_id, risk_score, trigger_
                                     signals (list of strings), explanation
                                     (plain text), regulatory_citation
                                     (clause text + source), timestamp
  GET  /api/permits                -> active permits with zone_id, type,
                                     start/end time
  POST /api/simulate/advance       -> advances the synthetic data
                                     simulation by one tick (useful for
                                     demo control — lets us "step through
                                     time" live on stage)

Return JSON in exactly this shape from both the mock and the real
implementation so swapping one for the other requires zero frontend
changes. If either of us needs to change this contract later, we say so
explicitly and update this list before writing code.

GOAL FOR THIS SESSION: acknowledge you understand this scope and the API
contract above. Do not start writing code until I give you a specific
numbered prompt from our build plan — I will paste those one at a time.
```

## Track A — backend / agents / RAG (Person A, branch `feature/backend-agents`)

### A1 — synthetic data generator (days 3–4)

```
Build the synthetic data generator for our project (see shared context
above). Create a Python module `data_sim/generator.py` that produces:

1. Gas/IoT sensor readings: for each of 5 plant zones, a time series of
   gas concentration (ppm), updated per simulated "tick", that normally
   fluctuates within a safe band but can be told to trend upward toward a
   configurable LEL (lower explosive limit) threshold over N ticks — we
   need this so we can script a demo scenario on stage.
2. Permit records: hot work, confined space entry, and general maintenance
   permits, each with zone_id, type, start_time, end_time, and a flag for
   whether a fire watch is logged.
3. Maintenance/shift logs: scheduled maintenance tasks per zone with a
   due date and completion status (so we can flag "overdue"), and shift
   roster entries with start/end times per zone (so we can flag "long
   consecutive shift" as a fatigue-risk proxy).

Requirements:
- Everything must be deterministic-but-configurable via a random seed, so
  we can reproduce the same demo scenario every time we rehearse.
- Include one pre-built "demo scenario" function that scripts exactly the
  compound-risk situation we want to show live: a hot work permit opens
  in zone 3 at tick 10, gas in zone 3 starts trending upward at tick 12,
  no fire watch is logged, and by tick 20 the combination should be
  clearly flaggable — but no single signal alone crosses a hard threshold
  before tick 18. This gap is what proves our fusion logic adds value
  over single-sensor alerting.
- Expose a simple `advance_tick()` function and a `get_current_state()`
  function returning all current sensor/permit/maintenance data as plain
  Python dicts — this is what the FastAPI layer will wrap next.
- Add a short README section explaining this is synthetic data and why
  (no real plant access), and how the demo scenario is scripted.

Do not build the FastAPI layer yet — just the generator module and a
small script that runs it for 30 ticks and prints the state, so I can
sanity-check the numbers before we wire it into the API.
```

### A2 — FastAPI ingestion layer (day 5)

```
Now wrap the data generator from A1 in a FastAPI app in `backend/main.py`.

Implement:
- GET /api/zones — return the 5 zones with a name and normalized (x, y)
  coordinates in a 0-1000 x 0-1000 space (we'll map this onto an SVG
  floorplan on the frontend)
- GET /api/permits — current active permits from the generator
- POST /api/simulate/advance — calls advance_tick() and returns the new
  raw state (not yet the risk score — that comes in A4)
- CORS enabled for local frontend dev (allow http://localhost:5173 and
  http://localhost:3000)

Keep the raw sensor/permit/maintenance state in memory (a module-level
dict is fine for the hackathon). Write a short curl-based smoke test
list in comments at the top of main.py so I can verify each endpoint
manually.

Match the response shapes to the API contract in the shared context
exactly, even though risk-scores and alerts aren't computed yet — return
them as empty lists/objects for now so Person B can start building the
frontend against this immediately.
```

### A3 — three domain agents (days 6–7)

```
Build the three domain agents as separate, independently-testable Python
modules under `backend/agents/`:

1. `gas_agent.py` — given a zone's gas reading history (last N ticks),
   return a finding: is it within normal range, trending up, or over
   threshold? Include the trend rate, not just a boolean, since we need
   this for the "how many minutes before critical" lead-time metric.

2. `permit_agent.py` — given active permits and current zone conditions,
   flag dangerous simultaneous operations: specifically hot work permits
   without a logged fire watch, and any permit active in a zone where the
   gas agent reports "trending up" or worse. Return the specific permits
   involved so the orchestrator can cite them by ID in the explanation.

3. `maintenance_agent.py` — flag overdue maintenance tasks and shift
   patterns exceeding a configurable consecutive-hours threshold (fatigue
   proxy).

Each agent should have a single main function taking the current state
(from A1/A2) and returning a small structured finding object: zone_id,
signal_type, severity (0-1), and a short plain-English description of
what it found. Keep these agents simple and rule-based/statistical —
do not call the LLM inside these three; the LLM reasoning belongs only
in the orchestrator (A4), so these stay fast and cheap to run every tick.

Write a quick test script that runs all three agents against the A1 demo
scenario and prints their findings tick-by-tick, so I can confirm the
individual signals stay "benign-looking" until the compound condition
should fire.
```

### A4 — orchestrator / fusion agent (days 8–9)

```
Build the orchestrator in `backend/agents/orchestrator.py`. This is our
core differentiator, so give it real attention.

Input: the findings from gas_agent, permit_agent, and maintenance_agent
for a given zone and tick.

Logic:
1. First pass without an LLM: implement an explicit rule that raises the
   compound risk score sharply when specific combinations co-occur (e.g.
   permit_agent flags "hot work without fire watch" AND gas_agent reports
   "trending up" in the same zone) even though neither alone crossed a
   hard threshold. Compute a 0-100 score per zone as a weighted function
   of the individual severities plus a co-occurrence bonus.
2. Second pass: call the Claude API (model claude-sonnet-4-6) with the
   three agents' findings as structured input, and ask it to produce a
   short plain-English explanation string of WHY the score is what it is
   — referencing the specific permit ID, gas trend, and any maintenance/
   shift factor involved. This explanation is what gets shown in the
   alert panel on the frontend, so it needs to read like something a
   safety officer would actually find useful, not generic.
3. Expose GET /api/risk-scores (current score per zone) and GET /api/
   risk-scores/history?zone_id=&hours= (time series, so add this to
   main.py's in-memory store as history accumulates across ticks) and
   GET /api/alerts (any zone currently above a configurable alert
   threshold, with the explanation from step 2 and a placeholder
   regulatory_citation field we'll fill in during A5).

Also compute and log a "lead time" number: how many ticks/minutes before
any single agent alone would have crossed its own hard threshold did the
compound score cross the alert threshold? We want this as a headline
number in the demo ("flagged 40 minutes before a single-sensor system
would have").

Wire this into main.py so all the API contract endpoints now return real
computed data instead of placeholders.
```

### A5 — regulatory RAG (days 10–12)

```
Build the regulatory RAG module in `backend/rag/`.

1. Corpus: gather 5-10 real, short clauses/sections from OISD guidelines,
   DGMS regulations, or the Factory Act 1948 / OSH Code 2020 that are
   relevant to hot work permits, confined space entry, and gas hazard
   management. Store each as a small text file or JSON entry with a
   clear citation label (document name + clause number). Also write 10-15
   short synthetic near-miss report snippets that reference similar
   compound conditions to what our demo scenario produces.
2. Embed this corpus into Chroma (or FAISS) using a standard embedding
   approach.
3. Build a retrieval function that, given an orchestrator alert's
   description (from A4), retrieves the most relevant clause(s) and near-
   miss report(s), then calls Claude to produce a short (1-2 sentence)
   grounded statement like: "This combination is addressed by OISD-XXX
   [paraphrased clause], and matches the pattern in near-miss report
   #4." Do not have the model invent a citation — only ever use the ones
   actually retrieved.
4. Wire this into the /api/alerts endpoint so each alert's
   regulatory_citation field is now populated by this real retrieval
   instead of a placeholder.

Keep the corpus small and hand-picked rather than trying to scrape a
large regulatory corpus — for the demo we need 100% grounded, correct
citations on the handful of alerts we'll actually trigger, not broad
coverage.
```

```
since we're heading into final integration.
```
